Rick & Morty API

The ground-floor reference implementation for the k3s-pi5 DevOps learning platform. A complete multi-service architecture — Node.js, .NET, PostgreSQL — built around a free, public, zero-auth REST and GraphQL API.

Reference Architecture · Encyclopedia

Every DevOps curriculum needs a sample application — something real enough to demonstrate production patterns, but simple enough that the infrastructure stays the focus, not the app logic. The Rick and Morty API was chosen as the external data source for this reference implementation because it hits a rare sweet spot:

What makes it ideal for a DevOps learning platform

PropertyWhy it matters for learning
Free & public No API keys, no billing, no sign-up. You can focus on architecture, not credential management. Every example in this project is reproducible without registering for anything.
REST + GraphQL Teaches both paradigms. The reference uses REST for simplicity, but GraphQL is available as a natural extension (see "Extending the Setup" in the README).
Stable data The show ended production in 2022. Characters, locations, and episodes are canonical — no drift, no breaking changes, no deprecations. Perfect for caching demos and CI/CD integration tests.
Predictable pagination 20 results per page, 42 pages of characters. Ideal for building background sync workers, demonstrating pagination patterns, and testing bulk-import strategies.
Rich relational data Characters link to episodes, locations have residents, episodes have cast lists. A natural fit for teaching SQL joins, junction tables, and full-text search indexes.
Memorable & fun DevOps documentation is often dry. The Rick and Morty framing makes the stack more approachable and gives every component a personality — Rick is the .NET layer, Morty is the Node.js frontend, the garage is PostgreSQL.

What this is NOT

  • Not a production app. This is a reference architecture for learning. It demonstrates patterns you use in production, but it is intentionally minimal.
  • Not a Rick and Morty fan site. The show is the theme, not the product. The product is the infrastructure patterns.
  • Not the only sample in k3s-pi5. This is the first reference. Later modules add sample apps in Go, Python, and .NET that build on the same architecture patterns.

The reference implements a 3-tier architecture with a .NET API gateway as the central orchestrator. Every data request from the frontend passes through the .NET layer — never directly to the upstream API or the database.

Key Architectural Decisions

DecisionRationale
.NET as the orchestrator The project author's primary stack is .NET/C#. The API gateway pattern maps naturally to ASP.NET Core's middleware pipeline — caching, rate limiting, auth, and logging are all first-class middleware concerns. The same patterns work in Express.js, Go, or Python — .NET is the example, not the requirement.
PostgreSQL over SQLite SQLite would be simpler but teaches nothing about connection strings, network isolation, or persistent volumes. PostgreSQL forces you to think about these — exactly the skills you need for Kubernetes StatefulSets.
Node.js frontend, not .NET Blazor A polyglot stack demonstrates cross-service communication patterns. The Node.js app must understand HTTP clients, error handling, and retries — universal skills independent of language.
Cache-aside, not write-through The API data is read-heavy and changes rarely. Cache-aside (check cache → miss → fetch → store → return) is the simplest pattern that still teaches cache invalidation and TTL management.
Fixed-window rate limiter Simple to understand, built into .NET 8 with zero extra packages, and produces clear 429 responses with Retry-After headers. A token-bucket or sliding-window limiter would be more precise but adds complexity without adding learning value at this stage.

The reference architecture is presented as four progressive scenarios. Each one adds a layer without removing the previous one — you see why each layer exists before it's added.

# Scenario What it teaches When to use Status
A Direct Fetch (no cache) HTTP client basics, JSON deserialization, API proxying Local dev, demos, freshness-critical Done
B .NET Proxy + PostgreSQL Cache Cache-aside pattern, TTL management, DB as L2 cache, connection pooling Production-like, reducing upstream calls Done
C Full Sync (bulk import) Background jobs (IHostedService), bulk inserts, paginated API walking, cron scheduling High-traffic, search, analytics, offline mode Done
D Containerized on k3s VM → Kubernetes mapping, Deployments, Services, Ingress, StatefulSets, PVCs, NetworkPolicies The target architecture — everything beyond local dev Done

The pedagogical thread

Scenario A is the "hello world" — it proves connectivity. Scenario B adds caching and introduces the database. Scenario C removes the upstream dependency entirely. Scenario D maps the whole thing to Kubernetes. Each step is small enough to understand in isolation, but together they cover the full journey from localhost to a production-shaped homelab cluster.

Every pattern in this reference implementation recurs in later k3s-pi5 modules. Understanding them here means you recognise them everywhere.

1. Cache-Aside Pattern

Client → API → Cache? ──hit──→ return cached
                    │
                   miss
                    │
                    ▼
              Upstream API → Store in cache → return

Where it recurs: M2 Docker Compose (.NET + Redis), M3 Sample Apps (Node.js + Postgres on k3s).

2. Rate Limiting (Fixed Window)

Request → RateLimiter middleware
              │
              ├─ within limit → controller action
              │
              └─ exceeded → 429 Too Many Requests + Retry-After header

Where it recurs: M3 Security (NetworkPolicies + RBAC), M4 Observability (Prometheus metrics on 429 counts).

3. Health + Metrics Endpoints (Un-rate-limited)

GET /health   → { "status": "healthy" }         ← k8s liveness/readiness
GET /metrics  → { "requests_served": 1423, ... }  ← Prometheus scrape

Where it recurs: Every module that deploys to k3s. M4 Observability replaces the stub with real Prometheus metrics.

4. Background Sync Worker

IHostedService → ExecuteAsync → while (!stopping)
    ├─ Wait(cron interval)
    ├─ Walk all API pages (1..42)
    ├─ Bulk insert/upsert into Postgres
    └─ Log sync summary

Where it recurs: M5 CI/CD (scheduled GitHub Actions), Phase 2 ArgoCD sync loops.

5. Environment-Driven Configuration

Every tunable — cache TTL, rate limit threshold, connection string, sync schedule — is an environment variable. No magic numbers in code. This pattern is non-negotiable and enforced across every k3s-pi5 module.

The .NET API enforces rate limiting using ASP.NET Core 8's built-in System.Threading.RateLimiting — zero extra NuGet packages.

Why rate limiting matters in THIS project

  • Protects the upstream API. The Rick and Morty API has no auth and no documented rate limit. Being a good API citizen means self-throttling.
  • Prevents runaway frontends. A buggy React effect loop could fire thousands of requests. The rate limiter catches this at the gateway before it reaches the database or upstream.
  • Demonstrates middleware composition. Rate limiting is the first middleware in the pipeline. It shows how ASP.NET Core's middleware order matters — auth, rate limiting, caching, then the controller.

Policy Design

Policy NameLimitWindowQueueApplies To
fixed 100 requests 60 seconds 0 (reject immediately) All /api/* endpoints
sync 2 requests 10 minutes 0 POST /api/sync/run only
none Disabled via [DisableRateLimiting] /health, /metrics

Why two policies?

Full syncs (Scenario C) walk 42 API pages and bulk-insert thousands of rows. They're expensive. A separate, stricter policy means someone can browse the API normally while the sync endpoint stays protected from accidental double-triggers. Health and metrics endpoints are deliberately excluded — Kubernetes probes fire every 10 seconds and Prometheus scrapes every 15 seconds; throttling either would trigger false-positive pod restarts.

Recommended Learning Path

  1. Read this encyclopedia page — understand the architecture, the decisions, the patterns.
  2. Read the README — study the full code, environment variables, and deployment options.
  3. Follow the k3s-pi5 modules in order (M1 → M2 → M3 → M4 → M5) on the main site.
  4. Return here whenever you encounter a pattern you recognise — cache-aside, rate limiting, health checks — and see how it fits into the larger architecture.

The Rick and Morty framing isn't just decorative — it gives every architectural component a personality that makes the stack easier to reason about:

CharacterMaps ToWhy
Rick Sanchez .NET API Layer Hyper-intelligent, does all the heavy lifting, owns the logic, talks to the outside world. Builds portal guns from scrap — like a well-factored middleware pipeline.
Morty Smith Node.js Frontend What users actually see. Anxious but functional. Inseparable from Rick — the frontend can't function without the API. His brainwaves cancel out Rick's genius — the UI simplifies the backend's complexity for human consumption.
The Garage PostgreSQL Where everything gets stored between adventures. Messy, chaotic, but contains every artifact from every timeline — like a database with full history and full-text search.
Portal Gun HTTP / REST Instant transport between dimensions — like an API call that fetches data from any endpoint in the universe.
Council of Ricks Kubernetes Cluster Multiple Ricks coordinating across dimensions. Each one is autonomous but governed by shared rules — like pods in a cluster managed by the control plane.
